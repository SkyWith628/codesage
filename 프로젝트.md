## 1. 요구사항 분석

### 한 문장 정의
> **"GitHub PR(Pull Request)이 열리면, 변경된 코드를 자동으로 분석해 인라인 리뷰 코멘트와 요약을 달아주는 AI 에이전트 서버"**

### CodeRabbit 핵심 기능 분해 (벤치마킹 대상)

| 기능 | 설명 |
|---|---|
| ① PR 요약 | 변경 내용을 자연어로 요약 + 파일별 변경표 |
| ② 인라인 리뷰 | 특정 코드 줄에 직접 코멘트 (버그/스타일/보안) |
| ③ 정적 분석 통합 | Linter 결과를 LLM 리뷰와 결합 |
| ④ 증분 리뷰 | 새 커밋이 추가되면 "그 부분만" 다시 리뷰 |
| ⑤ 대화형 후속 | 코멘트에 답글 달면 AI가 대응 |
| ⑥ 학습/컨벤션 | 레포 규칙·과거 리뷰를 반영 |

### MoSCoW 우선순위

```
Must  (반드시) — ① PR 요약, ② 인라인 리뷰, Webhook 수신/코멘트 게시
Should(해야함) — ③ 정적 분석 통합, ④ 증분 리뷰
Could (가능하면)— ⑤ 대화형 후속, ⑥ 컨벤션 학습(RAG)
Won't (안함)   — 멀티 언어 IDE 플러그인, 자체 결제 시스템
```

> **1차 목표는 Must + Should.** ⑤⑥은 확장 포인트로만 설계.

## 2. 시스템 설계 (Trade-off)

### 2-1. 전체 아키텍처

```
┌──────────┐   PR 이벤트(Webhook)   ┌─────────────────────────────┐
│  GitHub  │ ─────────────────────> │   API Server (FastAPI)      │
│          │ <───────────────────── │   - 서명 검증                │
│  PR 코멘트│   코멘트 게시(REST)     │   - 이벤트 라우팅             │
└──────────┘                        └───────────┬─────────────────┘
                                                 │ 작업 등록(빠른 응답)
                                                 ▼
                                    ┌─────────────────────────────┐
                                    │   Message Queue (Redis)     │
                                    │   - PR 리뷰 작업 대기열       │
                                    └───────────┬─────────────────┘
                                                 │ 작업 소비
                                                 ▼
                          ┌──────────────────────────────────────────┐
                          │   Worker (리뷰 엔진)                       │
                          │  1) Diff 수집  ──> 2) 청킹/필터링          │
                          │  3) 정적분석(Linter)                       │
                          │  4) 컨텍스트 수집(관련 파일/컨벤션)         │
                          │  5) LLM 리뷰 (Claude API)                  │
                          │  6) 코멘트 게시                            │
                          └──────────┬───────────────────┬───────────┘
                                     ▼                   ▼
                          ┌──────────────┐    ┌─────────────────────┐
                          │  PostgreSQL  │    │  Claude API         │
                          │  (리뷰 기록)  │    │  (LLM 추론)          │
                          └──────────────┘    └─────────────────────┘
```

### 2-2. 핵심 설계 결정 4가지

#### 결정 ① 동기 처리 vs **비동기 큐 처리** ← 선택

```
선택: Webhook은 즉시 200 응답 + 실제 리뷰는 백그라운드 Worker에서 처리

선택 이유:
  - GitHub Webhook은 10초 안에 응답 없으면 실패로 간주하고 재시도함
  - LLM 리뷰는 수십 초~수 분 걸림 → 동기로 하면 무조건 타임아웃

대안: 동기 처리 → 타임아웃·재시도 폭탄으로 운영 불가

Trade-off:
  얻는 것: 안정성, 확장성(Worker 수만 늘리면 됨), 재시도 제어
  잃는 것: 인프라 복잡도↑ (Redis + Worker 프로세스 추가 관리)

면접 설명: "Webhook의 10초 제약과 LLM의 긴 추론 시간이 충돌하므로,
            큐를 둬서 수신과 처리를 분리(decoupling)했습니다"
```

#### 결정 ② 전체 파일 투입 vs **Diff 중심 + 선택적 컨텍스트** ← 선택

```
선택: 변경된 Diff(hunk)를 중심으로 보내고, 필요한 주변 코드만 추가

선택 이유:
  - 전체 레포를 LLM에 넣으면 토큰(비용) 폭증 + 컨텍스트 한계 초과
  - 리뷰는 본질적으로 "바뀐 부분"이 대상

Trade-off:
  얻는 것: 비용 절감, 속도, 컨텍스트 한계 회피
  잃는 것: 파일 간 의존성은 놓칠 수 있음 → Should 단계의 RAG로 보완
```

#### 결정 ③ LLM 단독 vs **정적 분석 + LLM 하이브리드** ← 선택

```
선택: Linter/타입체커로 1차 필터링 → LLM은 "의미/설계" 리뷰에 집중

선택 이유:
  - "세미콜론 누락, 미사용 변수"는 Linter가 0원에 정확히 잡음
  - 역할 분담: 기계적 규칙=Linter, 맥락적 판단=LLM

Trade-off:
  얻는 것: 비용↓, 정확도↑ (확실한 건 결정론적 도구가 처리)
  잃는 것: Linter 통합/파싱 코드 추가 (언어별 어댑터 필요)
```

#### 결정 ④ LLM 선택 — **Claude (Anthropic API)**

```
선택: claude-opus-4-8 (정밀 리뷰) + claude-haiku-4-5 (요약/분류)

선택 이유:
  - 긴 코드 컨텍스트 처리와 코드 이해 품질이 높음
  - 모델 라우팅으로 작업 난이도별 비용 최적화
  - Prompt Caching으로 반복 시스템 프롬프트 비용 절감
```

## 6. 핵심 코드 뼈대

### 6-1. Webhook 서명 검증 (`app/core/security.py`)

```python
import hashlib
import hmac

def verify_github_signature(
    payload_body: bytes,
    signature_header: str | None,
    secret: str,
) -> bool:
    """GitHub Webhook의 HMAC 서명을 검증한다."""
    if signature_header is None:
        return False

    expected = "sha256=" + hmac.new(
        key=secret.encode("utf-8"),
        msg=payload_body,
        digestmod=hashlib.sha256,
    ).hexdigest()

    # ★ 타이밍 공격 방지: 일반 == 비교 금지
    return hmac.compare_digest(expected, signature_header)
```

```
[What] GitHub가 보낸 요청이 진짜인지 비밀키로 검증
[Why]  /webhook은 공개 엔드포인트 → 가짜 이벤트로 LLM 비용 폭증 가능.
       compare_digest는 비교 시간이 일정해 타이밍 공격을 막음.
[How]  모든 Webhook 수신 서버(Stripe/GitHub/Slack)의 표준 보안 패턴.
```

### 6-2. Webhook 라우터 (`app/api/webhook.py`)

```python
from fastapi import APIRouter, Request, Header, HTTPException
from app.core.config import settings
from app.core.security import verify_github_signature
from app.queue.producer import enqueue_review_job

router = APIRouter()

@router.post("/webhook")
async def github_webhook(
    request: Request,
    x_github_event: str = Header(...),
    x_hub_signature_256: str | None = Header(None),
):
    raw_body = await request.body()   # 서명 검증은 '원본 바이트'로

    if not verify_github_signature(raw_body, x_hub_signature_256, settings.WEBHOOK_SECRET):
        raise HTTPException(status_code=401, detail="Invalid signature")

    payload = await request.json()

    if x_github_event == "pull_request" and payload["action"] in ("opened", "synchronize"):
        await enqueue_review_job(
            repo=payload["repository"]["full_name"],
            pr_number=payload["pull_request"]["number"],
            head_sha=payload["pull_request"]["head"]["sha"],
            action=payload["action"],   # opened=전체, synchronize=증분
        )

    return {"status": "accepted"}   # ★ 즉시 응답 → 타임아웃 회피
```

```
[What] 서명 검증 후 리뷰 작업을 큐에 '등록만' 하고 즉시 응답
[Why]  여기서 LLM을 직접 돌리면 GitHub 타임아웃 → 큐에 던지고 손 뗌.
       action=="synchronize"로 '새 커밋 push'도 잡아 증분 리뷰 처리.
[How]  Producer-Consumer 패턴 — 수신과 처리의 분리.
```

### 6-3. 리뷰 파이프라인 (`app/review/pipeline.py`)

```python
from app.review import diff_parser, static_analysis, context_builder
from app.review import llm_reviewer, comment_poster
from app.integrations.github_client import GitHubClient

async def run_review(repo: str, pr_number: int, head_sha: str, action: str) -> None:
    """PR 리뷰 6단계 파이프라인을 순서대로 실행한다."""
    gh = GitHubClient(repo)

    # 1) Diff 수집 — 증분이면 '직전 리뷰 이후 변경분'만
    diff = await diff_parser.fetch_diff(gh, pr_number, incremental=(action == "synchronize"))

    # 2) 청킹 + 노이즈 파일 필터링
    chunks = diff_parser.chunk_and_filter(diff)
    if not chunks:
        return

    # 3) 정적 분석
    lint_findings = static_analysis.run_linters(chunks)

    # 4) 컨텍스트 수집 (Should)
    context = await context_builder.build(gh, chunks)

    # 5) LLM 리뷰 (구조화 JSON)
    review = await llm_reviewer.review(chunks, lint_findings, context)

    # 6) 게시
    await comment_poster.post(gh, pr_number, head_sha, review)
```

```
[What] 6단계(수집→청킹→정적분석→컨텍스트→LLM→게시)를 한 흐름으로 지휘
[Why]  각 단계를 독립 모듈로 분리 → 단위 테스트 쉽고 단계별 교체 가능.
[How]  ML/데이터 파이프라인 표준 구조. 단일 책임 원칙(SRP)으로 유지보수성 확보.
```

### 6-4. LLM 리뷰어 (`app/review/llm_reviewer.py`)

```python
import json
from anthropic import AsyncAnthropic
from app.core.config import settings

client = AsyncAnthropic(api_key=settings.ANTHROPIC_API_KEY)

SYSTEM_PROMPT = """You are a senior code reviewer.
Review ONLY the provided diff. Classify each issue by severity
(critical/warning/suggestion) and category (bug/security/performance/style).
Return STRICT JSON only, no prose."""

async def review(chunks, lint_findings, context) -> dict:
    user_content = _build_prompt(chunks, lint_findings, context)

    resp = await client.messages.create(
        model=settings.REVIEW_MODEL,   # claude-opus-4-8
        max_tokens=4096,
        system=[{
            "type": "text",
            "text": SYSTEM_PROMPT,
            "cache_control": {"type": "ephemeral"},   # ★ 프롬프트 캐싱
        }],
        messages=[{"role": "user", "content": user_content}],
    )
    return _parse_json(resp.content[0].text)   # 환각/형식오류 방어 파싱
```

```
[What] diff+linter+컨텍스트를 프롬프트로 만들어 Claude에 보내고 구조화 JSON 수신
[Why]  자연어 줄글은 줄 번호를 못 붙임 → JSON 강제로 정확한 인라인 게시 가능.
       시스템 프롬프트 캐싱으로 반복 호출 비용 절감.
[How]  LLM을 '대화'가 아니라 '함수'처럼 쓰는 Structured Output 패턴.
```

## 8. 보안 체크

```
□ Webhook 서명 검증 (HMAC) — 가짜 이벤트 LLM 비용 폭탄 차단  ✅ 설계됨
□ Rate Limiting — 단시간 과도 PR 이벤트 throttle
□ 비밀 정보 환경변수화 — API키/시크릿/토큰은 .env, Secrets Manager
□ 최소 권한 GitHub App — 'PR 읽기 + 코멘트 쓰기'만 (코드 push 권한 ❌)
□ 프롬프트 인젝션 방어 — PR 코드 내 악성 지시문 격리
□ 민감정보 로깅 금지 — diff 안의 API키/비밀번호 로그 미기록
□ LLM 출력 검증 — suggested_code 자동 적용 ❌, '제안'으로만 게시
```

> **면접 차별점:** "AI 에이전트가 외부 입력(PR 코드)을 LLM에 넣을 때 프롬프트 인젝션을 어떻게 막았나"는 LLM Agent 직무 단골 질문.

## 9. 배포 전략

```
개발/로컬 : docker-compose (api + worker + redis + postgres)

운영:
  ┌ API 서버  → ECS (상시 떠 있어야 Webhook 수신)
  └ Worker    → ECS (큐 길이 기반 오토스케일링)
  ┌ Redis     → ElastiCache (관리형)
  └ Postgres  → RDS (관리형)

CI/CD:
  GitHub Actions: push → 테스트 → Docker 빌드 → ECR 푸시 → ECS 롤링 배포

왜 Lambda가 아닌 ECS?
  - 리뷰가 수 분 걸려 Lambda 15분 제한·콜드스타트 부담
  - 상시 Worker로 큐를 꾸준히 소비하는 모델이 안정적
  → 트래픽이 매우 간헐적이면 Lambda+SQS도 대안
```

## 10. 활용 방법

### 👤 관점 A — 일반 개발자 (설치 후 아무것도 안 해도 됨)

```
1. 평소처럼 코드 작성 → git push → PR 생성
        ▼
2. 1~2분 뒤, PR에 codesage-bot 자동 등장:

   ┌─ PR 상단 (요약 코멘트) ────────────────────────────┐
   │ 🧙 CodeSage Review                                  │
   │ ## 변경 요약                                         │
   │ 로그인 API에 JWT 인증을 추가했습니다. 3개 파일 변경.  │
   │ | 파일 | 변경 내용 |                                  │
   │ | auth.py | JWT 발급/검증 로직 추가 |                 │
   │ 발견: 🔴 Critical 1 · 🟡 Warning 2 · 💡 Suggestion 3│
   └─────────────────────────────────────────────────────┘

   ┌─ 코드 23번째 줄 (인라인 코멘트) ──────────────────────┐
   │ 🔴 [Security] JWT 시크릿이 하드코딩되어 있습니다.      │
   │    💡 제안:                                            │
   │    - SECRET = "my-secret-key"                         │
   │    + SECRET = os.environ["JWT_SECRET"]                │
   └───────────────────────────────────────────────────────┘
        ▼
3. 제안 보고 → 수정 → 다시 push
        ▼
4. CodeSage가 '새로 바뀐 부분만' 다시 리뷰 (증분 리뷰)
   → 고친 게 맞으면 이전 코멘트는 resolved
```

핵심: **개발자가 배워야 할 명령어가 없다.** PR을 평소처럼 쓰면 끝.

### 🛠️ 관점 B — 도입하는 사람 (한 번만 설정)

```
[1단계] GitHub App 등록
  - GitHub Settings → Developer settings → New GitHub App
  - 권한: Pull requests = Read & Write, Contents = Read only
  - 이벤트 구독: Pull request
  - Webhook URL: https://your-server.com/webhook
  - Webhook secret: 랜덤 문자열 → .env의 WEBHOOK_SECRET에 저장

[2단계] 서버 실행
  $ cp .env.example .env        # 비밀값 채우기
  $ docker compose up -d        # api + worker + redis + postgres 기동

  # .env 주요 값:
  ANTHROPIC_API_KEY=sk-ant-...
  WEBHOOK_SECRET=...
  GITHUB_APP_ID=...
  GITHUB_PRIVATE_KEY=...

[3단계] 레포에 설치
  - CodeSage App 페이지 → Install → 리뷰받을 레포 선택
  - 끝. 그 레포의 모든 PR이 자동 리뷰됨
```

### ⚙️ 관점 C — 팀 리드 (리뷰 정책 커스터마이징)

`config/codesage.yaml` 한 파일로 팀 규칙 주입:

```yaml
review:
  model: "claude-opus-4-8"          # 정밀 리뷰용
  summary_model: "claude-haiku-4-5" # 요약은 싼 모델

  focus:          # 중점 관점 (팀 우선순위)
    - security
    - bug
    - performance
    - style

  ignore:         # 리뷰 노이즈 제거
    - "*.lock"
    - "dist/**"
    - "**/migrations/**"

  min_severity: "warning"   # suggestion(사소) 코멘트 끔

  guidelines: |   # 팀 컨벤션을 자연어로 주입
    - 모든 API 함수에 type hint 필수
    - print() 대신 logger 사용
    - DB 쿼리는 반드시 parameterized query
```

```
[What] YAML 한 파일로 "무엇을, 얼마나 깐깐하게, 무엇을 빼고" 리뷰할지 제어
[Why]  팀마다 규칙이 다름 → 설정 분리로 비개발자도 규칙 변경 가능
[How]  CodeRabbit/ESLint 표준 — '코드 ≠ 설정' 분리
```

## 14. 용어 사전

| 용어 | 한국어 | 설명 |
|---|---|---|
| PR (Pull Request) | 풀 리퀘스트 | 내 변경을 메인 브랜치에 합쳐달라는 요청 |
| Webhook | 웹훅 | 이벤트 발생 시 외부 서버로 보내는 자동 HTTP 알림 |
| Diff / Hunk | 디프 / 헝크 | 변경 전후 코드 차이 / 연속 변경 덩어리 한 블록 |
| Queue | 큐, 대기열 | 작업을 순서대로 쌓아 하나씩 처리하는 자료구조 |
| Decoupling | 디커플링 | 두 모듈이 직접 의존하지 않게 분리 |
| Token | 토큰 | LLM이 텍스트를 처리하는 최소 단위 (비용 기준) |
| RAG | 검색 증강 생성 | 관련 문서를 검색해 프롬프트에 끼워넣는 기법 |
| Linter | 린터 | 코드를 실행 않고 규칙 위반을 잡는 정적 분석 도구 |
| Hallucination | 환각 | LLM이 사실 아닌 내용을 그럴듯하게 지어냄 |
| HMAC | 해시 기반 인증 코드 | 비밀키로 메시지 위변조를 검증하는 서명 |
| Prompt Injection | 프롬프트 인젝션 | 입력에 악성 지시를 숨겨 LLM을 조종하는 공격 |
| Structured Output | 구조화 출력 | LLM 응답을 정해진 형식(JSON)으로 강제 |
| Producer-Consumer | 생산자-소비자 | 작업 등록과 처리를 큐로 분리하는 패턴 |
| SRP | 단일 책임 원칙 | 하나의 모듈은 하나의 일만 책임 |
| ngrok | 엔그록 | 로컬 서버를 임시 공개 URL로 터널링 |
